Component Architecture

SourceScribe Core Component Architecture

SourceScribe is a powerful documentation generation tool that automatically creates user-friendly technical documentation from source code. The core of the system is composed of several interconnected modules that work together to analyze code, generate diagrams, and produce the final documentation.

Component Breakdown​

The SourceScribe core is divided into the following major components:

API Providers​

The API providers are responsible for interfacing with various AI/ML services to generate content and diagrams. The current providers include:

  • AnthropicProvider: Integrates with the Anthropic API for natural language processing and generation.
  • OLLaMAProvider: Integrates with the Anthropic OLLaMA language model for more advanced text generation.
  • OpenAIProvider: Integrates with the OpenAI API for natural language processing and generation.

These providers are abstracted behind a common BaseProvider interface, allowing the system to easily swap out or add new providers as needed.

Configuration​

The configuration system is responsible for loading and parsing the SourceScribe configuration file, which defines the project structure, documentation settings, and other options.

  • Loader: Loads the configuration file and validates the settings.
  • Models: Defines the data models for the various configuration options.

Engine​

The engine is the core of the SourceScribe system, responsible for analyzing the source code, generating diagrams and content, and producing the final documentation.

  • Analyzer: Analyzes the source code and extracts relevant information.
  • FeatureGenerator: Generates the content and diagrams for the documentation.
  • Diagram: Handles the creation of various types of diagrams (sequence, class, state, etc.).
  • Generator: Orchestrates the overall documentation generation process.

Utilities​

The utility modules provide various helper functions and classes used throughout the SourceScribe system.

  • FileUtils: Provides file-related utility functions.
  • GitHubUtils: Handles interactions with GitHub, such as generating permalinks.
  • Logger: Provides a centralized logging system.
  • Parser: Handles parsing of various data formats.

Watch​

The watch system is responsible for monitoring the source code for changes and triggering the documentation generation process.

  • Handler: Handles file change events and initiates the documentation generation.
  • Watcher: Monitors the source code directory for changes.

Communication Patterns​

The CLI initiates the documentation generation process, which then flows through the various components. The Analyzer examines the source code, the FeatureGenerator uses the API providers to generate content and diagrams, and the Diagram component renders the visual elements. The Generator orchestrates the overall process and provides the final documentation to the CLI.

Data Models​

The SourceScribe core utilizes several key data models to represent the various configuration options and metadata:

The SourceScribeConfig represents the overall project configuration, while the ModuleConfig and DiagramConfig models capture the details of individual modules and diagrams, respectively.

Design Patterns​

SourceScribe utilizes several design patterns to achieve a modular, extensible, and maintainable architecture:

  1. Provider Pattern: The API providers are abstracted behind a common BaseProvider interface, allowing the system to easily swap out or add new providers as needed.
  2. Factory Pattern: The Factory module is responsible for creating the appropriate API provider based on the configuration, decoupling the provider selection from the rest of the system.
  3. Dependency Injection: The various components are loosely coupled, with dependencies injected as needed, making the system more testable and easier to extend.
  4. Separation of Concerns: The system is divided into distinct components, each with a clear and focused responsibility, promoting modularity and maintainability.
  5. Configuration-Driven Design: The system is heavily driven by the configuration, allowing users to customize the behavior and output without modifying the core codebase.

Extension Points​

SourceScribe is designed to be easily extensible, with several key extension points:

  1. API Providers: New API providers can be added by implementing the BaseProvider interface and registering them with the Factory.
  2. Configuration: The configuration system allows users to customize the project structure, include/exclude patterns, and other settings to fit their specific needs.
  3. Diagram Types: New diagram types can be added by implementing the Diagram interface and registering them with the DiagramFactory.
  4. Content Generation: The FeatureGenerator can be extended to add new content generation capabilities, such as support for additional programming languages or documentation formats.
  5. Utilities: The various utility modules, such as FileUtils and GitHubUtils, can be extended or replaced to integrate with custom tooling or workflows.

By leveraging these extension points, users and contributors can tailor SourceScribe to their specific requirements and integrate it into their existing development and documentation processes.
